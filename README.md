# Paper Insight

Paper Insight 是一个本地优先的论文推荐 Web 应用，用来持续同步 arXiv 最新论文，并通过配置的大模型接口对候选论文做分析、打分和总结。

默认主题聚焦“大模型/智能体 + 网络自治/网络数字孪生/系统架构”的研究方向，包括网络自治、自智网络、零接触网络、网络数字孪生、意图网络、LLM Agent、多智能体协同、Agent 工程化集成、闭环自治和面向网络的智能体系统。`5G/6G`、无线网络、网络通信、异常检测、流量预测、根因分析等更具体的领域或业务/运维场景仍可在查询构建器里手动勾选，但不再作为默认强制条件。

## 当前能力

- 从 arXiv RSS 自动同步最新论文到本地论文库。
- 从本地论文库按时间范围和查询条件筛选候选论文。
- 用户确认候选论文后，再逐篇调用所选大模型分析。
- 分析失败后支持从当前失败论文继续重试，已成功分析的论文不会重复调用模型。
- 高于推荐阈值的论文进入推荐列表，其他论文进入隐藏列表；低于阈值的论文不会为了凑数被补进推荐。
- 支持设置“最低达标数”：首轮达标论文超过该数时全部保留；首轮不足时自动启动下一轮候选查询和分析，达到最低达标数后停止追加分析。每轮候选数、推荐列表目标和后端分析上限都最高可设为 100。
- 支持针对单个推荐列表生成“精选论文阅读清单”：先选择周报候选子集，再基于论文原文或摘要重新复评、筛选和生成 Markdown，便于手动发布到洞察网站。
- 支持论文探索页，把历史生成过的论文集中浏览。
- 支持摘要翻译，翻译必须通过 LLM API。
- 支持强制使用 arXiv API 重新获取候选，并在界面中标注候选来源。

## 核心流程

1. 在左侧确认或编辑论文搜索条件。
2. 点击“生成推荐列表”。
3. 流程弹窗展示候选获取进度。
4. 候选列表出现后，用户确认要分析的论文。
5. 可选择“强制 arXiv API 重新获取”，绕过本地库直接查询 arXiv API。
6. 所选大模型逐篇分析论文，界面展示当前论文、耗时和进度。
7. 如果首轮阈值以上论文数量低于“最低达标数”，应用会从下一批候选开始新一轮查询和分析；一旦达到最低达标数，就停止继续分析追加候选，而不是从低分论文里补足。每轮最多可取 100 篇候选，推荐列表目标最多 100 篇，自动补轮最多 10 轮。
8. 如果某篇论文分析失败，点击“重试分析”会从失败的当前论文继续，不重做已经成功的论文。
9. 分析完成后生成新的推荐列表。
10. 打开某个推荐列表后，可以点击“准备生成周报”，先从该列表中选择周报候选子集，再基于论文原文重新评分和筛选，最后生成发布版 Markdown。
11. 可以在推荐模式、探索模式和单篇分析页之间切换。

## 论文来源

候选论文只来自 arXiv。默认路径是：

```text
arXiv RSS -> 本地 arXiv 库 -> 候选筛选 -> 用户确认 -> 大模型分析
```

强制获取路径是：

```text
arXiv API -> 候选列表 -> 用户确认 -> 大模型分析
```

界面会标注每篇候选论文的来源：

- `本地 arXiv 库`：来自后端 RSS 同步后的本地论文库。
- `arXiv API`：来自 `export.arxiv.org/api/query` 的直接查询。
- `arXiv API 缓存`：来自直接 API 查询后的短期缓存或异常兜底。

更完整的论文获取、RSS 同步、本地库、429 处理和大模型输入说明见 [论文获取逻辑说明](PAPER_FETCHING_README.md)。

## 后端自动同步

后端进程会自动维护 arXiv RSS 同步，不依赖系统定时器调用 HTTP 接口。

同步判断逻辑：

- 后端读取 `.cache/arxiv-papers.json` 里的 `lastSyncedAt`。
- 按 `lastSyncedAt + ARXIV_DAILY_SYNC_MS` 计算下一次同步时间。
- 如果服务停了几天，重启后发现本地库过期，会自动同步。
- 如果刚同步过，会等到真正到期再同步。
- 同步失败后会按 `ARXIV_AUTO_SYNC_RETRY_MS` 重新尝试。

默认 RSS 分类：

```text
cs.NI, cs.AI, cs.LG, cs.MA, cs.DC, cs.IT, eess.SP, eess.SY
```

## 大模型分析

论文推荐必须使用配置的大模型接口。应用不会使用本地关键词规则兜底。

推荐分析阶段的大模型输入包括论文标题、作者、分类、日期、arXiv 链接和摘要。生成发布版阅读清单时，服务端会优先抓取 arXiv HTML 原文，清洗并截取摘要、方法、系统、实验、结果、局限和结论等关键段落，先给周报候选重新生成四维分数和总分，再按周报入选阈值筛选并生成 Markdown；如果某篇论文没有可用 HTML 原文，会自动降级为摘要和已有分析复评，不会中断整份周报。

大模型负责生成：

- 分维度评分和兴趣适配分类
- 一句话概要
- 问题、背景、方法、技术细节、贡献、实验、网络价值和局限分析
- 阅读路径和推荐理由
- 摘要翻译

服务端负责基于四维分项重新计算推荐总分，不直接采用模型返回的总分。这样可以保证摘要初筛、周报复评和前端展示使用同一套权重与校准逻辑。

如果 LLM 调用失败、结果缺字段、超时或没有返回某篇论文的分析，服务端会返回错误。应用不会自动无限重试，用户可以在界面上手动确认是否重试；重试会保留已成功分析的结果，并从当前失败论文继续。LLM 返回 JSON 时如果在合法 JSON 后附带少量解释文本，服务端会抽取第一段完整 JSON 对象继续处理；如果 JSON 本身不完整或缺少论文结果，仍会报错。

## 评分维度

当前使用 4 个研究质量维度辅助打分：

- 研究问题价值：20%
- 方法新意：30%
- 系统价值：20%
- 证据强度：30%

网络、AI、ICT、电信或 ADN 方向匹配不混入四维质量分，避免把“题材相关”误判成“研究质量高”。服务端会先按研究问题、方法新意、系统价值和证据强度计算研究质量分，再用兴趣适配做最终推荐分校准：网络自治/电信方向小幅上浮，通用 AI/Agent 方法轻微上浮，专用非目标领域轻度降权，方向不明只轻微后移。兴趣适配只做加减分，不做封顶。

服务端总分计算逻辑：

```text
base = 研究问题价值 * 0.20 + 方法新意 * 0.30 + 系统价值 * 0.20 + 证据强度 * 0.30
balancePenalty = max(0, base - min(方法新意, 证据强度)) * 0.12
weakEvidencePenalty = max(0, 70 - 证据强度) * 0.20
researchQuality = clamp(base * 1.20 - 22 - balancePenalty - weakEvidencePenalty, 0, 100)
最终推荐分 = clamp(researchQuality + interestAdjustment, 0, 100)
```

兴趣适配使用 4 个档位：

| 档位 | 校准 |
| --- | --- |
| 网络自治/电信方向 | `+4`。包括 ADN、O-RAN、5G/6G、网络数字孪生、意图驱动、闭环自治、网络运维、路由/QoS/频谱/切片/故障诊断等。 |
| 通用 AI/Agent 方法 | `+2`。包括不绑定专用行业的 LLM/Agent/RAG/工具调用/评测/安全/系统架构等通用方法。 |
| 专用非目标领域 | `-6`。主问题、评价对象或主要应用场景明确落在医学、生命科学、脑科学、基因组、地理、游戏、教育、金融、法律、社科、推荐系统等专用垂直领域，且摘要没有展示可迁移的通用 AI/Agent/系统方法时使用。若只是使用垂直数据做验证，但主要贡献是通用机制、架构、评测或 agent 方法，则归入通用 AI/Agent 方法。 |
| 方向相关性不明 | `-2`。没有明确目标方向，也看不出通用可迁移方法时使用；这只是轻微后移，不等同于非目标。 |

这意味着方法新意和证据强度仍然是主要拉开差距的质量维度；方向只做最终排序校准。只有业务方向匹配但方法机制、系统设计或证据支撑不足的论文，不会因为 ICT、ADN 或电信场景相关就自动进入高分区；专用非目标领域论文会排后，但质量特别强时仍可能进入 70 分左右的扫读区。

总分档位：

| 分数段 | 含义 |
| --- | --- |
| 0-49 | 基本不达标。概念模糊、贡献弱、证据不足，或只是产业/实践/流程性内容。 |
| 50-59 | 弱相关或弱贡献。有一定问题意识或应用场景，但研究设计、方法机制或证据明显不足。 |
| 60-69 | 一般候选。方向或问题有价值，但创新、系统完整性、证据至少有一项明显短板。 |
| 70-79 | 值得扫读。问题清楚，有一定研究贡献或系统价值，但还不是强论文。 |
| 80-89 | 重点关注。问题、方法、系统或证据中至少两项比较强，值得深入读。 |
| 90-100 | 少数高价值论文。问题重要，方法有实质贡献，证据扎实，结论边界清楚，值得跟踪或复现。 |

分维度评分也使用同样档位：研究问题价值看问题是否清晰、重要、可验证；方法新意看是否有可复用的新机制、建模、任务定义或评测协议；系统价值看解决框架的模块、接口、数据流、闭环机制、部署约束和可复用性；证据强度看实验、数据、基线、消融、泛化、鲁棒性和复现线索。若只能基于摘要和元数据判断，模型必须保守评分并说明限制。

推荐分低于 60 的论文必须给出“不推荐原因”，但它不是分数解释。原因不能写“总分低于 60”“某维度多少分”或模板化阅读建议，而要像评审意见一样指出这篇论文具体差在哪里：例如方法只是流程拼装、缺少可复用机制，系统接口/闭环/失败处理没说清，实验缺少数据集/基线/消融/真实场景，或摘要只能看到愿景而看不到可验证假设。

推荐分 70 及以上的论文必须给出显性的“高分信号”。一句话描述不能只是复述论文做了什么，而要说明为什么它比普通候选更值得读：核心贡献是什么，最强维度来自研究问题、方法新意、系统价值还是证据强度，以及有哪些可核验的方法、系统或证据信号。前端会在高分论文卡片和详情页展示这条高分信号。

`ICT` 只在论文主问题域确实属于通信/电信/网络基础设施时展示，例如 5G/6G、RAN/O-RAN、无线/蜂窝/移动/核心/边缘/光/卫星网络、网络切片、路由、QoS、频谱、切换、业务保障、告警关联或故障诊断。泛 AI agent、泛多智能体、泛 graph/neural network、社交网络或只出现 `network` 一词，不会自动打 `ICT` 标签，也不会获得网络自治/电信方向加分。

评分规则版本变更时，前端会自动清空浏览器中已有的推荐报告和打分结果，避免旧规则分数与新规则混用。查询条件、API Key 和模型设置不受影响。

## 推荐展示

推荐报告包含三种视图：

- 推荐论文：分数达到阈值的论文。
- 隐藏论文：分数低于阈值的论文，仍然可以打开查看，但不会计入最低达标数。
- 全部分析：本次已分析的所有论文。

单篇论文分析页用于连续阅读，会保留大模型生成的技术分析正文，不再展示摘要和维度条。

论文探索页会汇总历史推荐列表里的论文，方便跨列表浏览和搜索。

## 阅读清单

每个推荐列表都可以生成一份“精选论文阅读清单”。周报生成是独立功能，和原始推荐列表之间只传递论文子集：原列表的推荐/隐藏、旧分数、旧排序和旧评价不会被修改，也不会直接决定周报排序。

生成阅读清单前，可以先调三个周报专用参数：

- 复评候选下限：只用于从原始列表中取一组论文子集，便于把摘要初筛时低于推荐线但仍有潜力的文章纳入复评。界面范围为 `0-95`，默认 `60`，按 `5` 分步进。
- 周报入选阈值：服务端抓取原文并重新给出四维分数后，优先选择复评分达到该阈值的论文。界面范围为 `40-95`，默认沿用推荐阈值或 `70`，按 `5` 分步进。
- 保底入选篇数：如果达到周报入选阈值的论文不足，服务端会按复评分从高到低补足到保底数量。界面范围为 `1-20`，默认 `3`；补入论文仍保留真实分数和短板，周报中只能作为快速扫读或补充观察，不包装成强推荐。

周报候选最多提交 100 篇。服务端会为周报候选尝试抓取 arXiv HTML 原文并做本地缓存，然后基于原文摘录重新评价研究问题价值、方法新意、系统价值和证据强度，得到周报专用的新总分；同时会要求模型根据作者区、机构脚注、邮箱域名、致谢、项目页或摘要上下文提取发表单位线索，并将机构名称翻译成中文。抓取失败的论文会降级为摘要和已有分析复评，并在周报生成提示中保留依据边界。周报复评产生的新分数、排序、发表单位线索和入选结果只用于本次周报，不回写原始推荐列表。

周报生成阶段包括：

```text
选择候选 -> 提交任务 -> 抓取原文 -> 周报复评 -> 生成 Markdown -> 接收结果 -> 保存到当前列表
```

如果没有论文满足“复评候选下限”，前端会提示先调低候选下限；如果达到“周报入选阈值”的论文少于保底数量，服务端会按复评分补足到可用候选数和保底数量中的较小值。

阅读清单由 LLM 生成 Markdown，标题格式为：

```text
【精选论文】YY年M月第N周阅读清单：一句话观点
```

输出内容包括：

- YAML 前置元数据，包含 `title` 和 `description`，便于发布到洞察网站。
- 正文一级标题必须直接使用完整标题：`【精选论文】YY年M月第N周阅读清单：一句话观点`。冒号后的观点控制在 18-32 个中文字符以内，要提炼本周核心趋势或观点，避免复述标题或空泛表述。
- 报告导读，概括本周高价值论文、特别值得关注的文章，以及对华为 ADN 网络研究最有价值的方向。
- 本周趋势判断，提炼多篇论文共同呈现出的研究趋势、成熟度风险和对 ADN 的关联。
- 本周必读、值得跟进、快速扫读三个层级。
- 每篇论文的中文发表单位、周报复评分、符合的评分维度、内容方法与结果、ADN 启发与阅读价值、局限与适用约束。
- 推荐阅读顺序和完整论文清单；完整清单使用“论文、一句话介绍、阅读级别、链接”，不重复展示发表单位、周报复评分或符合维度，也不展示内部阈值或方向列。
- 最后一行由服务端固定补齐发布署名、开源联系和工具/模型说明。

## API Key

API Key 可以在页面左上角齿轮设置里输入。Key 只保存在当前浏览器会话的 `sessionStorage`，不会写入项目文件。

也可以通过环境变量提供大模型接口配置：

```powershell
$env:LLM_API_KEY="your-api-key"
$env:LLM_MODEL="your-model"
$env:LLM_API_URL="https://your-llm-provider.example/api"
node server.js
```

常用环境变量：

- `LLM_API_KEY`
- `LLM_MODEL`
- `LLM_API_URL`
- `LLM_MAX_OUTPUT_TOKENS`：默认 `12000`，控制 LLM 单次分析的最大输出 token。
- `LLM_RESPONSE_MAX_CHARS`：默认 `500000`，只做超大响应保护；不会把模型返回内容截断成短文本。
- `LLM_REQUEST_TIMEOUT_MS`：默认 `600000`，控制 LLM 分析请求超时时间。

## 配置

```powershell
$env:PORT=3000
$env:HOST="127.0.0.1"
```

`HOST` 或 `BIND_HOST` 用来控制监听地址。远端部署建议设置为 `127.0.0.1`，避免 Web 页面和 API 直接暴露在公网。

```powershell
$env:ARXIV_MIN_INTERVAL_MS=3500
$env:ARXIV_CACHE_TTL_MS=1800000
$env:ARXIV_STALE_CACHE_TTL_MS=86400000
$env:ARXIV_429_COOLDOWN_MS=1800000
$env:ARXIV_429_MAX_COOLDOWN_MS=7200000
$env:ARXIV_DAILY_SYNC_MS=72000000
$env:ARXIV_AUTO_SYNC=1
$env:ARXIV_AUTO_SYNC_INITIAL_DELAY_MS=30000
$env:ARXIV_AUTO_SYNC_RETRY_MS=3600000
$env:ARXIV_RSS_CATEGORIES="cs.NI,cs.AI,cs.LG,cs.MA,cs.DC,cs.IT,eess.SP,eess.SY"
```

## 本地运行

项目不依赖第三方 npm 包，使用 Node.js 原生 HTTP 服务。

```bash
node server.js
```

打开：

```text
http://localhost:3000
```

## 远端部署

远端建议使用 systemd 常驻运行，并让服务只监听远端本机：

```ini
Environment=NODE_ENV=production
Environment=PORT=3000
Environment=HOST=127.0.0.1
ExecStart=/usr/bin/node /home/<user>/paper-insight/server.js
```

远端不直接开放公网 `3000` 端口。需要访问远端页面时，在本机开启 SSH 隧道：

```powershell
& 'D:\Program Files\Git\usr\bin\ssh.exe' -i D:\code\paper-insight\.cache\paper_insight_ed25519 -N -L 3001:127.0.0.1:3000 <user>@<server>
```

然后打开：

```text
http://localhost:3001
```

常用远端命令：

```bash
sudo systemctl status paper-insight
sudo systemctl restart paper-insight
sudo journalctl -u paper-insight -f
```

## 注意事项

- API Key 不应该提交到仓库。
- 本地缓存保存在 `.cache/`，该目录不会提交到 Git。
- 强制 arXiv API 可能遇到 `429 Rate exceeded`，这是 arXiv 服务端限流，不代表应用逻辑卡死。
- 远端部署时不要把服务直接绑定到 `0.0.0.0` 暴露公网，优先使用 SSH 隧道访问。
