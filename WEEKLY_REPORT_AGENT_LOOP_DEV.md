# 周报生成 Agent Loop 需求与开发设计

本文档用于讨论和设计“周报生成”功能的下一阶段优化。范围仅限周报生成链路，不改变原始推荐列表的评分、排序、隐藏/推荐状态，也不修改摘要初筛的历史结果。

## 目标

把当前“一次复评 + 一次成稿”的周报生成流程，升级成可观察、可校准、可恢复的受控 Agent Loop：

1. 先围绕每篇论文建立更可靠的证据上下文。
2. 再基于全文或摘要做周报专用复评。
3. 然后在候选论文之间做横向校准，拉开分数和阅读优先级。
4. 最后生成报告，并由质量检查环节发现格式、逻辑和推荐强度问题。

核心原则是“让 Agent 多走几步，但每一步都受规则约束”。不要让系统变成完全自由发挥的自治代理。

## 当前基线

当前周报生成已经具备这些能力：

- 原始推荐列表只提供论文子集，周报阶段重新评价，不回写原列表。
- 可配置复评候选下限、周报入选阈值和保底入选篇数。
- 服务端优先抓取 arXiv HTML 原文，失败后降级为摘要和已有分析。
- 周报复评使用四维分数：研究问题价值、方法新意、系统价值、证据强度。
- 最终报告标题包含一句短观点，正文使用固定页尾说明。

主要短板：

- 复评虽然独立，但仍偏“一次性打分”，缺少跨论文校准，所以分数容易挤在同一区间。
- 生成报告时缺少显式的“证据卡片”，容易写得像摘要转述，而不是精选阅读判断。
- 低优先级或保底补入论文的短板解释仍可能不够具体。
- 长链路中间状态不可复用，失败后通常需要重新跑完整流程。
- 质量检查主要依赖提示词约束，缺少独立校验和修正步骤。

## 产品需求列表

Agent Loop 不是一个单独展示给用户的功能，而是一种实现方式。下面的表先按用户可理解的需求来拆，再说明“当前普通方法”和“Agent Loop 方法”的区别。

| 优先级 | 需求 | 输入 | 当前普通方法 | Agent Loop 方法有什么不同 | 输出 |
| --- | --- | --- | --- | --- | --- |
| P0 | 准备周报候选 | 原始推荐列表、复评候选下限、是否使用原文 | 从原列表按现有分数筛一批论文，直接进入生成流程 | 把原列表只当候选池，明确记录候选来源、旧分数边界、原文可用性和后续处理计划 | 周报候选集合、候选数量、上下文来源状态 |
| P0 | 基于全文重新复评 | 候选论文、原文/摘要、评分标准 | 一次性让模型给每篇论文重新打分和写理由 | 先提取论文证据，再按证据逐篇评分；评分理由必须对应问题、方法、实验、结果或局限，不能只解释分数 | 周报四维分数、总分、具体复评理由 |
| P0 | 横向校准和拉开差距 | 所有候选论文的复评结果 | 每篇论文独立打分，分数容易集中在 70 分以上 | 把同一批论文放在一起比较，重新校准方法新意、证据强度和系统价值；识别“方向匹配但研究质量一般”的论文 | 校准后分数、阅读层级、排序依据 |
| P0 | 周报入选决策 | 校准后分数、入选阈值、保底篇数 | 达阈值就入选，不足时按分数补足 | 达阈值论文正常入选；保底补入论文必须带 fallback 标记，并在报告里低优先级呈现 | 最终入选论文、入选原因、fallback 标记 |
| P0 | 生成发布版周报 | 入选论文、校准分数、复评理由、论文依据 | 直接把入选论文交给模型生成 Markdown | 先基于本周论文提炼核心观点，再围绕“为什么值得读、读哪里、局限是什么”组织内容 | 可复制发布的 Markdown 周报 |
| P0 | 报告质量检查 | 生成后的 Markdown、格式规则、入选结果 | 主要依赖提示词一次生成正确 | 生成后再做一次检查，发现标题、页尾、旧分数泄漏、理由模板化、fallback 过度推荐等问题时定向修正 | 通过检查的最终 Markdown、检查结果 |
| P0 | Agent 步骤日志查看 | 周报任务各阶段的输入摘要、执行动作和阶段结果 | 用户只能看到整体进度和最终报告，难以判断多阶段处理带来的增益 | 每个 Agent 步骤都记录“做了什么、发现了什么、如何影响后续判断”，用户可展开查看；日志只用于审查和调试，不写入最终报告 | 可查看的步骤日志、关键中间结果摘要、失败定位 |
| P1 | 中间结果缓存与重试 | 候选论文、原文、证据卡片、复评结果 | 失败后大概率重新跑完整流程 | 每个阶段有可保存的中间结果，失败后从上一个成功阶段继续 | 可复用的阶段结果、失败定位 |
| P1 | 周报候选预检界面 | 候选列表和原文抓取预估 | 用户点击生成后才知道耗时或失败 | 生成前展示候选数量、原文可用数量、预估流程和风险 | 启动前确认信息 |
| P1 | 用户反馈记忆 | 人工校对意见、用户对高低分的反馈 | 每次评分都只依赖当前提示词 | 把用户修正沉淀成偏好规则，用于后续复评和校准 | 更贴近用户判断的评分偏好 |
| P2 | 主题聚类 | 入选论文、证据卡片 | 按论文逐篇排列 | 先归纳本周主题，再按主题组织报告结构和标题观点 | 更像趋势观察的周报结构 |

建议第一阶段只做 P0。P0 的目标不是增加很多按钮，而是把后台生成过程从“一次性调用”变成“可观察、可校准、可检查”的流程。P1 和 P2 等实际跑过几轮后再决定是否值得做。

## 推荐的 Agent Loop

周报生成可以拆成 8 个阶段：

1. 准备候选
   - 输入来自原始推荐列表的论文子集。
   - 只使用论文元数据、摘要、原文抓取结果和已有分析作为上下文。
   - 明确标记原始分数只作背景，不可用于周报排序。

2. 获取上下文
   - 优先读取 arXiv HTML 原文。
   - 抓取失败时降级为摘要。
   - 记录每篇论文的上下文来源：全文、部分全文、摘要降级。

3. 生成证据卡片
   - 每篇论文生成结构化 evidence card。
   - 字段包括研究问题、核心方法、实验设置、结果证据、局限、系统落地价值、方向标签。
   - 证据卡片不直接决定分数，但为后续复评提供依据。

4. 单篇复评
   - 基于证据卡片给四维分数。
   - 每个分数必须有具体档位理由。
   - `ICT`、电信、ADN、网络自治等方向匹配只作为标签或背景，不参与分数。

5. 横向校准
   - 把同一批候选论文放在一起比较。
   - 对方法新意和证据强度做强制排序或分层。
   - 识别“业务方向很匹配但论文研究质量一般”的论文，避免总分虚高。
   - 输出 calibrated score，而不是直接采用单篇复评原始分。

6. 入选决策
   - 优先选择达到周报入选阈值的论文。
   - 如果数量不足，按校准后分数补足保底数量。
   - 保底补入论文必须带 `fallback` 标记，并在报告中放到快速扫读或补充观察层级。

7. 报告生成
   - 基于入选论文、校准分数、证据卡片和入选理由生成 Markdown。
   - 标题格式为 `【精选论文】YY年M月第N周阅读清单：一句话观点`。
   - 报告重点不是复述摘要，而是说明为什么值得读、读哪里、局限是什么。

8. 质量检查与修正
   - 检查 YAML、标题、H1、页尾、章节结构。
   - 检查是否引用了原始推荐分数或旧排序。
   - 检查是否把方向匹配写成评分依据。
   - 检查低优先级论文是否给出具体短板。
   - 检查 fallback 论文是否被过度推荐。
   - 如失败，触发一次定向修正，而不是整篇重新生成。

## 四类 Agent 角色

第一阶段不需要真的做成多个并发 Agent，可以在服务端实现成四个顺序步骤：

| 角色 | 输入 | 输出 | 关键约束 |
| --- | --- | --- | --- |
| Evidence Extractor | 原文摘录、摘要、元数据 | 证据卡片 | 只抽取依据，不打总分 |
| Review Agent | 证据卡片、评分标准 | 四维分数、理由、阅读价值 | 不参考旧分数 |
| Calibration Agent | 全部复评结果 | 校准分数、分层、排序 | 拉开差距，解释调整原因 |
| Editor + QA Agent | 入选论文、证据卡片、校准结果 | 发布版 Markdown、检查结果 | 格式固定，内容具体 |

这样实现成本较低，也方便以后把某些步骤替换成更强的模型或更严格的本地规则。

## Agent 方案设计和技术选型

### 总体选型

第一阶段不建议引入复杂 Agent 框架。周报生成的流程边界很清楚，适合先用服务端受控 pipeline 实现：

- `Orchestrator` 负责串联步骤、记录状态、处理失败和重试。
- 每个 Agent 步骤实现成一个独立异步函数。
- 需要判断、总结和写作的步骤调用 LLM。
- 确定性规则交给服务端代码处理，例如阈值筛选、保底补入、标题格式、页尾检查和总分计算。
- LLM 输出尽量使用严格 JSON，服务端负责解析、字段校验、降级和日志记录。
- 中间结果优先保存在本地缓存中，先不引入数据库。

这样做的好处是开发复杂度低、可调试性强，也和现有 Node 服务端架构更一致。后续如果某个步骤变复杂，再考虑替换为更完整的 Agent 框架。

### 新增第三方依赖结论

第一阶段建议不新增第三方件。

这里说的 Agent Loop 不是引入某个外部 Agent 框架，而是把现有周报生成逻辑改成多阶段、可检查、可记录的服务端流程。也就是说：

- 不引入 LangChain、AutoGen、CrewAI 等 Agent 框架。
- 不引入新的数据库。
- 不引入新的任务队列系统。
- 不引入新的前端状态管理框架。
- 不引入新的模型 SDK，继续沿用现有服务端调用方式。

第一阶段主要复用现有能力：

| 能力 | 第一阶段选择 |
| --- | --- |
| 服务端流程控制 | 现有 Node 服务端中增加受控 pipeline |
| LLM 调用 | 复用当前服务端调用方式 |
| 原文抓取 | 复用现有 arXiv HTML 抓取和缓存逻辑 |
| 中间结果保存 | 先复用本地 `.cache` 文件 |
| 进度展示 | 复用现有请求状态接口，扩展阶段和步骤日志字段 |
| JSON 校验 | 第一阶段先用服务端代码做必要字段校验 |
| 报告格式检查 | 先用服务端规则检查标题、H1、页尾、表格和旧分数泄漏 |

后续只有在出现明确瓶颈时，再考虑新增依赖：

| 可能场景 | 再考虑的技术 |
| --- | --- |
| 阶段日志和中间结果越来越多 | 轻量数据库 |
| 多用户并发生成周报明显变多 | 后台任务队列 |
| JSON 结构越来越复杂 | JSON Schema 校验库 |
| Agent 工具调用路径变得开放且不可预测 | 专门的 Agent 框架 |

因此，当前开发重点不是“加三方件”，而是把现有流程重构成清晰的步骤：每一步有输入、输出、日志、校验和失败处理。

### 为什么第一阶段不直接使用 Agent 框架

不是因为 Agent 框架没有价值，而是当前周报生成问题更像“受控评审流水线”，不是开放式自治任务。

Agent 框架最适合这些场景：

- 工具很多，Agent 需要自己决定下一步调用哪个工具。
- 任务路径不固定，需要规划、执行、观察、再规划。
- 多个 Agent 需要长期协作、互相委派任务。
- 需要框架自带的 tracing、memory、guardrails、evals 或复杂工作流管理。

而第一阶段的周报生成路径比较固定：

```text
候选论文 -> 原文上下文 -> 证据卡片 -> 单篇复评 -> 横向校准 -> 入选决策 -> 报告写作 -> 质量检查
```

这里最难的不是“让 Agent 自己决定干什么”，而是：

- 每一步输入输出要稳定。
- 评分标准要固定。
- 日志要能被用户看懂。
- 旧分数不能污染新复评。
- 最终报告不能泄漏内部过程。
- 失败后要能定位具体阶段。

这些能力即使用 Agent 框架，也仍然需要自己设计 schema、提示词、校验逻辑、日志结构和 UI 展示。直接上框架反而可能带来额外成本：

| 风险 | 说明 |
| --- | --- |
| 抽象成本 | 框架概念可能比当前需求更重，调试链路更长 |
| 依赖成本 | 项目从轻量无依赖变成需要维护框架版本和兼容性 |
| 部署成本 | 远端服务部署、依赖安装和故障排查会变复杂 |
| 日志不匹配 | 框架 tracing 未必天然适合当前产品里的“步骤日志”展示 |
| 控制力下降 | 框架可能鼓励更开放的工具调用，但本功能需要严格边界 |

所以第一阶段建议先做自定义轻量 Agent Loop。为了不把后续路线堵死，可以在设计上保留一个简单的步骤接口：

```text
AgentStep
  name
  input
  run()
  validate()
  summarizeLog()
  output
```

如果后续发现确实需要更复杂的工具规划、多 Agent 协作或标准化 tracing，再把某些 `AgentStep` 替换成框架实现。这样既不牺牲当前开发效率，也保留未来升级空间。

### 开发语言和代码组织

第一阶段继续使用 JavaScript/Node.js 实现，不新增 Python 服务或其他运行时。

这里的“不换语言”不等于继续把所有逻辑都堆在 `server.js` 里。为了避免后续维护不方便，建议实现时把周报 Agent Loop 拆成独立模块，再由现有服务端入口调用。

建议代码组织：

```text
server.js
weekly-report/
  orchestrator.js
  context-builder.js
  evidence-extractor.js
  review-agent.js
  calibration-agent.js
  selection-engine.js
  report-writer.js
  qa-checker.js
  step-log.js
  schema.js
```

这样仍然是 JS/Node 项目，但每个文件只负责一类事情：

| 模块 | 作用 |
| --- | --- |
| `orchestrator.js` | 串联完整流程，控制阶段状态和失败处理 |
| `context-builder.js` | 抓取、清洗和缓存论文原文 |
| `evidence-extractor.js` | 调用 LLM 提取证据卡片 |
| `review-agent.js` | 调用 LLM 做单篇复评 |
| `calibration-agent.js` | 调用 LLM 做轻量横向校准 |
| `selection-engine.js` | 用确定性规则筛选入选论文和 fallback 论文 |
| `report-writer.js` | 调用 LLM 生成发布版 Markdown |
| `qa-checker.js` | 做格式和内容质量检查 |
| `step-log.js` | 记录每个阶段的输入摘要、输出摘要、影响和错误 |
| `schema.js` | 集中放结构校验、默认值和字段清洗 |

继续用 JS 的优点：

- 和现有项目一致，部署方式不变。
- 不需要维护第二套 Python 服务、环境变量和进程。
- Node 的异步模型适合抓原文、调用 LLM、轮询状态这类 I/O 流程。
- 前后端都用 JS，数据结构更容易对齐。

需要注意的点：

- 不要把新的 Agent Loop 继续写成一个超长函数。
- 不要让 LLM 步骤之间互相隐式依赖，必须通过明确 JSON 字段传递结果。
- 不要让前端直接理解复杂中间对象，前端只展示服务端整理后的步骤日志摘要。
- 如果未来需要大量离线评估、批量统计或复杂文本分析，可以再增加 Python 辅助脚本，但不作为第一阶段主链路。

### Orchestrator

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 不使用 |
| 职责 | 控制周报生成全流程，维护阶段状态、步骤日志、失败信息和最终输出 |
| 输入 | 用户参数、候选论文集合、当前报告上下文 |
| 输出 | 阶段状态、步骤日志、最终周报结果 |
| 技术实现 | 服务端异步 pipeline；每个阶段有明确 `pending/running/done/failed` 状态 |
| 日志重点 | 当前执行到哪一步、处理了多少篇、失败发生在哪个阶段、下一步是什么 |

Orchestrator 是这条链路的控制器，不应该参与论文判断。它的价值是把“一个黑盒生成请求”拆成用户可以观察的步骤。

### Context Builder

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 默认不使用 |
| 职责 | 获取 arXiv HTML 原文、清洗正文、截取关键段落，失败时降级为摘要 |
| 输入 | 论文元数据、arXiv 链接、摘要 |
| 输出 | 每篇论文的上下文文本、上下文来源、抓取状态 |
| 技术实现 | 沿用现有原文抓取与缓存逻辑；增加阶段结果摘要 |
| 日志重点 | 哪些论文拿到全文，哪些只有摘要，哪些抓取失败，以及失败原因 |

Context Builder 不负责判断论文好坏，只负责把后续 Agent 需要读的材料准备好。

### Evidence Extractor

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 使用 |
| 职责 | 从原文或摘要中提取论文证据卡片 |
| 输入 | 论文上下文、摘要、标题、作者、分类 |
| 输出 | 研究问题、核心方法、实验设置、结果证据、局限、系统价值、方向标签 |
| 技术实现 | 可以按论文逐篇处理，便于失败重试和日志展示；输出结构化 JSON |
| 日志重点 | 每篇论文提取到了哪些有效证据，是否存在证据不足或只能基于摘要判断 |

Evidence Extractor 不打分。它的目的不是生成漂亮文字，而是把“后面为什么这么评”需要用到的依据先固定下来。

### Review Agent

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 使用 |
| 职责 | 基于证据卡片给周报专用四维分数和具体理由 |
| 输入 | 证据卡片、评分标准、论文元数据 |
| 输出 | 四维分数、分项理由、阅读价值判断、不确定点 |
| 技术实现 | LLM 只输出分项分数和理由；总分由服务端按固定权重计算 |
| 日志重点 | 每个分数对应的具体依据，以及是否因为证据不足而保守评分 |

Review Agent 不能参考原始推荐列表的旧分数，也不能把 ICT、ADN、产业方向匹配当成加分项。方向匹配只能作为标签或背景。

### Calibration Agent

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 使用，但范围要克制 |
| 职责 | 在同一批候选中检查明显高估或低估，给出阅读层级和必要的小幅校准 |
| 输入 | 所有候选的证据卡片、复评分数、复评理由 |
| 输出 | 原始复评分、校准后分数、阅读层级、校准原因 |
| 技术实现 | 不做强制分布；只允许在发现明确相对误判时调整，并要求解释 |
| 日志重点 | 哪些论文被上调或下调，原因是方法、证据、系统价值还是局限 |

Calibration Agent 的目标不是让每期分数都按固定比例分布，而是避免同批论文里“工程总结”和“扎实研究”分数过近。为了保持不同周报之间可比较，建议同时保留：

- `rawReviewScore`：按固定评分标准得到的原始复评分。
- `calibratedScore`：用于本期周报排序的校准分。
- `readingTier`：本期阅读层级。
- `calibrationReason`：调整原因。

### Selection Engine

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 不使用 |
| 职责 | 根据阈值和保底数量决定最终入选论文 |
| 输入 | 校准后分数、入选阈值、保底篇数 |
| 输出 | 入选论文、入选方式、fallback 标记 |
| 技术实现 | 服务端确定性规则 |
| 日志重点 | 达阈值论文数量、保底补入数量、最低入选分 |

Selection Engine 必须保持规则透明。保底补入是为了保证周报可用数量，不代表论文被强推荐。

### Report Writer

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 使用 |
| 职责 | 生成发布版 Markdown 周报 |
| 输入 | 入选论文、证据卡片、复评分数、校准结果、阅读层级 |
| 输出 | Markdown 正文、标题观点、导读、逐篇点评、完整清单 |
| 技术实现 | LLM 写作，服务端后处理标题、H1、页尾和格式 |
| 日志重点 | 采用了什么本周核心观点，哪些论文支撑这个观点，fallback 论文如何呈现 |

Report Writer 不负责重新决定论文是否入选。它只负责把已经完成筛选和校准的结果写成适合发布的内容。

### QA Checker

| 项目 | 设计 |
| --- | --- |
| 是否使用 LLM | 混合使用 |
| 职责 | 检查发布内容是否符合格式、边界和质量要求 |
| 输入 | 生成后的 Markdown、入选结果、评分规则 |
| 输出 | 检查结果、问题列表、必要时的修正版 Markdown |
| 技术实现 | 代码规则优先检查确定性问题；LLM 辅助检查理由空泛、过度推荐和观点支撑不足 |
| 日志重点 | 发现了哪些问题，哪些由代码自动修正，哪些要求模型定向改写 |

QA Checker 可以拆成两层：

- 规则检查：标题格式、H1、页尾、Markdown 表格、是否出现旧分数、是否泄漏内部阈值。
- 内容检查：推荐理由是否具体、fallback 是否被包装成强推荐、标题观点是否能被入选论文支撑。

第一阶段可以先做规则检查和一次定向修正，不必做复杂多轮改写。

### 第一阶段实现顺序

建议按下面顺序实现，而不是一次性把所有 Agent 做完：

1. `Orchestrator` 和阶段日志。
2. `Context Builder` 阶段结果结构化。
3. `Evidence Extractor`。
4. `Review Agent`。
5. 轻量 `Calibration Agent`。
6. `Selection Engine` 日志化。
7. `Report Writer` 接入证据卡片和校准结果。
8. `QA Checker` 做规则检查和一次定向修正。

这个顺序的理由是：先让流程可观察，再提高评分和报告质量。否则如果先增加复杂评分逻辑，用户仍然只能看到最终报告，不容易判断 Agent Loop 的收益。

## 评分校准重点

这次优化最关键的是“横向校准”，否则只是在同一套提示词里多跑一次。

建议校准时加入这些规则：

- 一批候选中，`80+` 应该代表本周真正值得优先读的少数论文。
- `70-79` 是有明确阅读价值，但可能只在某一维特别强。
- `60-69` 是可扫读或背景补充，通常存在方法、证据或系统落地短板。
- `<60` 原则上不进入周报，除非保底机制需要补入，并且必须明确短板。
- 方法新意和证据强度不能只看作者声称，要看是否有实验、对比、消融、系统实现、公开数据或可复现实证。
- 方向匹配不能替代研究质量。产业相关、ICT相关、ADN相关只作为标签，不抬高总分。

建议额外记录两个分层字段：

- `readingTier`: `must_read` / `worth_reading` / `skim` / `background_only`
- `selectionReason`: `threshold` / `fallback`

这比只看总分更适合生成报告。

## 用户界面需求

第一阶段建议只做必要 UI，不扩张成复杂控制台：

- 生成前仍保留候选下限、入选阈值、保底数量。
- 增加“预计流程”展示：抓原文、证据提取、复评、校准、写作、质检。
- 生成过程中展示当前阶段和已完成数量。
- 增加“步骤日志”查看区，可以按阶段展开查看 Agent 执行过程和结果摘要：
  - 原文抓取阶段：展示哪些论文拿到了全文、哪些降级为摘要。
  - 证据提取阶段：展示每篇论文的问题、方法、实验、结果和局限摘要。
  - 复评阶段：展示四维分数、具体理由和不确定点。
  - 横向校准阶段：展示哪些论文被上调或下调，以及调整原因。
  - 质检阶段：展示发现了哪些格式或内容问题，以及是否已经修正。
- 步骤日志只在产品界面中用于审查和调试，不写入最终发布版 Markdown。
- 生成完成后展示周报入选摘要：
  - 候选总数。
  - 原文可用数量。
  - 达阈值数量。
  - 保底补入数量。
  - 最高分、最低入选分和分数分布。
- 报告正文仍保持可复制 Markdown，不把内部调试信息塞进发布内容。

## 验收标准

P0 阶段完成后，至少满足：

- 原始推荐列表中的旧分数、旧排序、推荐/隐藏状态不被修改。
- 周报生成可以清楚显示当前处于哪个阶段。
- 周报复评输出包含四维分数、总分、阅读层级和具体理由。
- 用户可以在生成界面查看每个 Agent 步骤的执行记录、关键发现和阶段结果摘要。
- 同一批 30 篇候选不应再出现绝大多数都高于 70 分且无法区分的情况。
- `ICT` 或产业方向匹配只能作为标签展示，不能作为高分理由。
- 低于 60 分或保底补入的论文必须说明具体短板，不能只说“总分低于阈值”。
- 最终 Markdown 标题、H1 和页尾符合既定格式。
- 最终 Markdown 不包含 Agent 步骤日志、内部调试信息、模型提示词或原始中间 JSON。
- 报告中不能泄漏内部阈值，也不能引用原始推荐列表旧分数。
- JSON 解析失败时能定位到具体阶段，并尽量保留已经完成的中间结果。

## 阶段拆分建议

第一期：

1. 增加周报任务状态机和阶段进度。
2. 增加证据卡片生成。
3. 增加单篇复评和横向校准。
4. 增加入选结果摘要。
5. 增加报告质量检查和一次定向修正。

第二期：

1. 缓存 evidence card、review 和 calibration 结果。
2. 支持失败后从上一个成功阶段继续。
3. 增加用户反馈记忆。
4. 优化周报候选预检界面。

第三期：

1. 加入主题聚类。
2. 加入报告改写循环。
3. 根据历史人工校对结果持续调整评分偏好。

## 风险和取舍

- 成本会上升：多阶段调用会增加等待时间和调用成本，需要缓存和候选数量控制。
- 结果更稳定但更慢：适合周报生成，不适合原始推荐列表初筛。
- 横向校准可能引入相对分布偏差：需要保留绝对评分标准，避免为了拉开差距而硬压分。
- UI 不能展示太多内部过程：用户需要知道系统在做什么，但发布内容应保持干净。
- 步骤日志需要控制粒度：展示关键判断和阶段结果，不展示完整原文、完整提示词、API Key 或过长的模型原始输出。

## 当前结论

最高优先级不是“让 Agent 更自由”，而是把周报生成变成一个可控的评审流水线：

候选论文 -> 原文/摘要上下文 -> 证据卡片 -> 单篇复评 -> 横向校准 -> 入选决策 -> 报告写作 -> 质量检查

这条链路最直接解决当前三个痛点：分数拉不开、理由不具体、报告像摘要堆叠。第一期建议只实现 P0，等实际跑过几轮周报后，再决定是否加入用户反馈记忆和主题聚类。
